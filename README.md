# Birthday Draw & Search

Session-based party game in a Drawful-ish direction:
- guests join a **session** via QR code
- players **draw prompts** on their phones
- afterwards they **search** for other drawings in the shared scene
- scores are tracked per session
- backend is split into **domain logic + repositories** so storage can be swapped later

## What changed in this refactor

- no more single global game state
- explicit **sessions** with `sessionId`
- session-specific HTTP and WebSocket flow
- asset URLs in game state instead of raw base64 blobs
- repository abstraction for
  - session persistence
  - asset persistence
- Cloud Run deployment scaffolding via `Dockerfile`

## Current storage implementation

Right now the repo ships with **local file-based repositories**:
- sessions are stored in `.data/sessions/<sessionId>.json`
- avatars and drawings are stored in `.data/assets/...`

That means:
- local development works nicely
- the code structure is now ready for Firestore / Cloud Storage adapters
- on Cloud Run this version should be run as **single instance** if you want consistent state, because the local filesystem is ephemeral

## Quick start

```bash
npm install
npm run dev
```

Then open:
- board: `http://localhost:5173/#/board`
- player: use the QR code generated by the board

## Production-ish local run

```bash
npm install
npm run build:all
npm run party
```

Then open:
- board: `http://localhost:3001/#/board`

The board creates a session automatically and shows a player QR code with the correct session link.

## HTTP API

- `POST /api/sessions` → create a new session
- `GET /api/sessions/:sessionId/state` → fetch current session state
- `POST /api/sessions/:sessionId/reset` → reset one session
- `GET /api/assets/...` → serve stored avatars and drawings

## WebSocket

Join message is now session-aware:

```json
{ "type": "join", "kind": "player", "sessionId": "abcd1234", "playerId": "optional-existing-player-id" }
```

## Cloud Run deployment

### Build container locally

```bash
docker build -t birthday-draw-search .
```

### Run container locally

```bash
docker run --rm -p 8080:8080 \
  -e PORT=8080 \
  -e DATA_ROOT=/tmp/birthday-data \
  birthday-draw-search
```

### Deploy to Cloud Run

```bash
gcloud run deploy birthday-draw-search \
  --source . \
  --region europe-west3 \
  --allow-unauthenticated \
  --timeout 3600 \
  --min-instances 1 \
  --max-instances 1
```

`max-instances=1` is intentional for the current file-based storage adapter.

## Next recommended step
